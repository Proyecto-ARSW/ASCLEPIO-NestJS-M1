Módulo Medicines — completado
Flujo: ADMIN crea/actualiza/desactiva medicamentos del catálogo. Todos los usuarios autenticados consultan. La baja es lógica (activo=false) para no romper historial de recetas.

categoria-medicamento.resolver.ts
medicines.module.ts — actualizado
Módulo Nurse — completado y corregido
Flujo: Enfermeros se crean vía /auth/register (rol ENFERMERO) o con createNurse vinculando un usuarioId existente. El módulo también gestiona su disponibilidad horaria (igual que médicos).

Archivos:

nurse.entity.ts, disponibilidad-enfermero.entity.ts
DTOs corregidos (create-nurse.input.ts ahora usa usuarioId como los médicos)
nurse.service.ts — corregido (había un h suelto en línea 31)
disponibilidad-enfermero.service.ts, disponibilidad-enfermero.resolver.ts
nurse.module.ts — actualizado
Módulo Inventario/Sedes — nuevo
Flujo: ADMIN crea sedes. ADMIN/RECEPCIONISTA registran/actualizan inventario. Al escribir stockActual, el trigger trg_disponibilidad_medicamento de PostgreSQL recalcula automáticamente disponibilidad (DISPONIBLE/STOCK_BAJO/AGOTADO) y actualizado_en — el backend no toca esos campos.

Archivos en src/inventario/

Módulo Historial Médico — nuevo
Flujo: MEDICO crea un registro de historial tras una consulta (puede vincularse a una cita con citaId). Solo el médico autor o ADMIN pueden modificarlo. El historial no se elimina (inmutabilidad clínica).

Archivos en src/historial/

Módulo Recetas — nuevo
Flujo: MEDICO crea recetas vinculadas a un historialId existente, referenciando un medicamentoId del catálogo. Las recetas no se eliminan; solo el ADMIN puede hacerlo en casos excepcionales.

Archivos en src/recetas/

Módulo Consentimientos — nuevo
Flujo: MEDICO/RECEPCIONISTA registran si el paciente otorgó consentimiento. Se puede adjuntar URL del documento firmado. Para retirar: revocarConsentimiento(id) → registra revocado=true y fechaRevocacion. Un consentimiento revocado no puede revocarse de nuevo.

Archivos en src/consentimientos/
